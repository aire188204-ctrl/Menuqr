================================================================================
STATE MACHINE & VALIDATION
================================================================================

Valid Transitions:

Pending          → Diagnosing           [Work starts]
Diagnosing       → Awaiting_Parts       [Need parts]
Diagnosing       → Completed            [Work done]
Awaiting_Parts   → Diagnosing           [Parts arrived]
Awaiting_Parts   → Completed            [Work done]
Completed        → Delivered            [Customer received]
Delivered        → (None)               [Terminal state]

Invalid transitions return 400 Bad Request with:
{
  "code": "INVALID_STATE_TRANSITION",
  "message": "Cannot transition from Completed to Pending",
  "details": {
    "invalid_transition": {
      "current_status": "Completed",
      "requested_status": "Pending",
      "valid_transitions": ["Delivered"]
    }
  }
}

================================================================================
COST CALCULATION
================================================================================

Formula: actual_cost = labor_fee + parts_total

parts_total = SUM(part[i].quantity × part[i].custom_price)

Examples:

1. Labor only:
   labor_fee = 200, used_parts = []
   → actual_cost = 200

2. Labor + parts:
   labor_fee = 150, used_parts = [{ qty: 2, price: 100 }, { qty: 1, price: 50 }]
   → parts_total = (2×100) + (1×50) = 250
   → actual_cost = 150 + 250 = 400

3. Donated parts:
   labor_fee = 75, used_parts = [{ qty: 1, custom_price: 0 }]
   → parts_total = 1 × 0 = 0
   → actual_cost = 75 + 0 = 75

All calculations are atomic - either fully calculated or transaction rolls back

================================================================================
RACE CONDITION PROTECTION
================================================================================

Problem: Two technicians complete repair simultaneously, both using same part

Solution: SELECT...FOR UPDATE (Row-Level Locking)

How it works:

  Tech A                              Tech B
  |                                   |
  |-- PATCH /status (qty=5)           |
  |   BEGIN TRANSACTION                |
  |   SELECT...FOR UPDATE              |
  |   (Locks row, available=10)        |
  |                                   |-- PATCH /status (qty=5)
  |                                   |   BEGIN TRANSACTION
  |                                   |   SELECT...FOR UPDATE
  |   Deduct: 10 - 5 = 5             |   (Waits for lock...)
  |   UPDATE status                    |
  |   COMMIT                           |
  |   (Releases lock)                  |
  |                                   |   SELECT...FOR UPDATE
  |                                   |   (Now sees available=5)
  |                                   |   Check: 5 < 5? NO
  |                                   |   Deduct: 5 - 5 = 0
  |                                   |   UPDATE status
  |                                   |   COMMIT

Result: Total deducted = 10 (correct), not 20 (would be wrong without locking)

If Tech B's requested qty was 6 (more than 5 available):
  → SELECT...FOR UPDATE gets current availability = 5
  → Check fails: 5 < 6
  → ROLLBACK transaction
  → Return 409 Conflict with { available: 5, requested: 6 }
  → Repair status remains unchanged (not Completed yet)
  → No inventory deducted

PERFORMANCE CHARACTERISTICS
================================================================================

Time Complexity:
  • Create job: O(1) - Single insert + UPSERT
  • Status update (no parts): O(1) - Single UPDATE
  • Status update (k parts): O(k) - k inventory checks + k deductions
  • Job number generation: O(1) - UPSERT on unique index

Expected Latencies:
  • Create job: 80-120ms
  • Status update (no parts): 100-150ms
  • Status update (1 part): 200-300ms
  • Status update (5 parts): 400-600ms

Bottlenecks:
  • Network round-trip to database (majority of time)
  • Row locking for inventory (brief, < 50ms each)
  • Parallel improvements: batch inventory checks (already done)

SUPPORT & TROUBLESHOOTING
================================================================================

Q: Job numbers have gaps (JOB-2026-0001, then 0003)?
A: Normal. Sequences are atomic but gaps occur when transactions rollback.

Q: Getting 409 Conflict but inventory shows plenty?
A: Concurrent repair just completed. Lock acquired by another transaction. Retry.

Q: Parts were deducted but status didn't update?
A: Transaction rolled back. Check audit_logs for error reason.

Q: Invalid state transition error for transition I think is valid?
A: Check REPAIR_API_GUIDE.md State Machine section. Some transitions aren't allowed.

Q: How do I track which technician completed the repair?
A: Check repairs.technician_id. Assign technician on create or via customer linking.

Q: Can I modify a repair after it's Delivered?
A: No. Delivered is terminal. System will reject status updates.

Q: How do I handle warranty on repairs?
A: Use the existing warranties table. Associate with repair via repair_id.
